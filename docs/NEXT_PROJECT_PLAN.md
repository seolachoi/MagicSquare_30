# 다음 프로젝트 개선 계획서

## 1. 목표

- 프롬프트 품질과 산출물 일관성을 높이고, 재작업 시간을 줄인다.
- 테스트 가능 요구사항(AC) 중심으로 개발 시작점을 통일한다.

## 2. 개선 전략

### 전략 A — 문제 정의 표준화

- 시작 단계에서 7-Step 또는 P-C-T-F 템플릿 적용
- 산출물: 문제정의 1p + 계약 정의 1p

### 전략 B — 품질 게이트 선적용

- 구현 전에 최소 RED 테스트 케이스 세트 정의
- 계약 변경 시 Traceability 업데이트 필수

### 전략 C — 기록 자동화

- 보고서/프롬프트 export 룰을 표준화
- 원문 transcript 보존을 기본값으로 설정

## 3. 실행 계획 (4주)

| 주차 | 실행 항목 | 담당 | 완료 기준 |
|---|---|---|---|
| 1주차 | 템플릿(문제정의/계약/AC) 확정 | PM/Tech Lead | 템플릿 리뷰 승인 |
| 2주차 | 파일 구조 및 백업 루틴 적용 | DevOps | 신규 프로젝트 100% 적용 |
| 3주차 | RED 선행 프로세스 파일럿 | Dev/QA | 파일럿 1건 완료 |
| 4주차 | 성과 측정 및 보정 | 전원 | 재작업 비율/리드타임 리포트 |

## 4. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 초기 문서 작성 부담 증가 | 일정 지연 | 템플릿 경량화(체크리스트 중심) |
| 팀별 문서 품질 편차 | 품질 불균형 | 예시 문서 + 리뷰 기준 제공 |
| 툴 인증/권한 문제 | 배포 지연 | Push 전 사전 체크리스트 의무화 |

## 5. 성공 지표

- 요구사항 변경으로 인한 재작업 횟수 30% 감소
- AC 기준 테스트 누락 0건
- 프로젝트 종료 시 회고/개선 문서 작성률 100%
