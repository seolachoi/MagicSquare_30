# Prompting BP 회고

## 1. Keep

- 문제 정의 → 계약 정의 → 테스트 관점으로 단계화한 접근이 요구사항 누락을 줄였다.
- 프롬프트에 제약조건(구현 금지, 코드 금지 등)을 명확히 적어 산출물 품질이 높아졌다.
- 산출물을 `Report/Prompt` 형태로 분리해 기록해 재사용성이 좋아졌다.

## 2. Problem

- 요청 범위가 길고 넓을 때 우선순위 합의 없이 진행되어 수정 반복이 발생했다.
- "전체 transcript" 요청 시 요약본과 원문 보존 기준이 섞여 혼선이 생겼다.
- GitHub 권한/인증 이슈가 후반에 발견되어 배포 완료 시간이 지연됐다.

## 3. Try

- 작업 시작 전 5분 범위 확정(이번 턴 목표, 산출물, 비범위)을 고정한다.
- Export 요청은 기본값을 "원문 전체 + 요약 섹션 선택"으로 표준화한다.
- Push 전 체크리스트(권한, 원격, 브랜치 추적, 인증 상태)를 선행한다.

## 6. 현업 적용 계획

- 요구사항 모호성이 큰 과제에는 7-Step 프레임워크를 표준 시작 템플릿으로 적용한다.
- 기능 구현 전 최소 1개 AC를 RED 기준으로 분해해 검증 가능성을 먼저 확보한다.
- 프로젝트 종료 시 회고 문서 + 액션 추적표를 릴리즈 체크리스트에 포함한다.
